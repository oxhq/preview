# Laravel Preview v0.1 Capture-to-Test Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` when available, or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the v0.1 `oxhq/preview` Laravel package that captures inbound traffic, verifies provider context, replays captures, generates fixtures, and generates Pest tests.

**Architecture:** Keep one Composer package with focused internal modules: Core, Capture, Providers, Testing, and optional Transports. v0.1 centers on capture-to-test; Route and Scenario modules are intentionally excluded.

**Tech Stack:** PHP 8.3+, Laravel package auto-discovery, Laravel Artisan commands, Pest-compatible generated tests, local filesystem storage, Symfony Process for tunnel/CLI adapters.

---

## Implementation Boundaries

Build in v0.1:

- package skeleton for `oxhq/preview`
- `PreviewServiceProvider`
- `config/preview.php`
- Core registry and safety services
- capture persistence
- provider model
- generic, HMAC, and Stripe providers
- exact and re-signed replay
- optional HTTP replay dispatch with `--send-to`
- Cloudflare Tunnel and ngrok transport adapters behind a registry
- fixture generation
- Pest test generation
- CLI commands listed in the canonical spec
- package HTTP capture endpoint at `/__preview/capture/{provider}`

Do not build in v0.1:

- dashboard UI
- hosted relay
- persistent URLs
- team auth
- cloud replay
- full route module
- full scenario module
- AST-based factory/test mutation
- database cloning
- package splitting

## File Map

Create these package files unless an existing implementation already provides an equivalent home:

- `composer.json` - package metadata, Laravel auto-discovery, dev dependencies.
- `src/PreviewServiceProvider.php` - config publishing and command registration.
- `config/preview.php` - storage, redaction, providers, transports, and live-mode defaults.
- `src/Core/*` - provider registry, transport contracts and adapters, capture IDs, redaction, safety results.
- `src/Capture/*` - request DTO, capture model, storage repository, replay service.
- `src/Providers/*` - provider interface, capability enum, verification result, generic/HMAC/Stripe providers.
- `src/Testing/*` - fixture writer, Pest test writer, generated fixture runtime helper.
- `src/Commands/*` - Artisan command classes for capture, capture:list, capture:show, capture:replay, capture:fixture, and capture:test.
- `tests/*` - package tests for provider behavior, capture storage, replay, generation, and command smoke.

## Task 1: Package Foundation

- [ ] Create `composer.json` with package name `oxhq/preview`, PHP `^8.3`, Laravel package auto-discovery, PSR-4 namespace `Oxhq\\Preview\\`, and test namespace `Oxhq\\Preview\\Tests\\`.
- [ ] Create `src/PreviewServiceProvider.php` that merges `config/preview.php`, publishes config, and registers v0.1 Artisan commands when running in console.
- [ ] Create `config/preview.php` with keys:
  - `storage_path`
  - `fixture_path`
  - `test_path`
  - `redact_headers`
  - `providers`
  - `transports`
  - `live_enabled`
- [ ] Add a package test bootstrap using Orchestra Testbench.
- [ ] Verify package discovery by running the package test suite with one smoke test that resolves the service provider.

## Task 2: Core Contracts And Registry

- [ ] Create `TunnelTransport` and `TunnelHandle` under `src/Core/Transport`.
- [ ] Create `TransportRegistry` and register `cloudflare` and `ngrok` adapters.
- [ ] Create process-backed Cloudflare Tunnel and ngrok adapters using Symfony Process, with fake-process tests instead of live binary requirements.
- [ ] Create `ProviderRegistry` that registers providers by name and throws a clear exception for unknown providers.
- [ ] Create `CaptureId` generation with sortable IDs or UUIDs; do not expose filesystem paths as capture IDs.
- [ ] Create `RedactionPolicy` that redacts `authorization`, `cookie`, `set-cookie`, and configured provider secret headers by default.
- [ ] Test provider lookup, missing-provider errors, capture ID generation, and header redaction.

## Task 3: Capture Storage

- [ ] Create `PreviewRequest` with method, path, query, headers, raw body, captured timestamp, provider name, and verification result.
- [ ] Create `CaptureRecord` with capture ID, provider, event type, endpoint, timestamps, redacted headers, raw-body path, and metadata.
- [ ] Create `CaptureRepository` that writes metadata as JSON and raw body as a separate local file under the configured storage path.
- [ ] Ensure raw capture storage is gitignored by default by documenting the generated storage path and creating package behavior that can append ignore guidance during install or first command run.
- [ ] Test storing, loading, listing, and showing captures without leaking redacted headers.

## Task 4: Providers

- [ ] Create `PreviewProvider`, `ProviderCapability`, and `VerificationResult`.
- [ ] Implement `GenericProvider` with capture, fixture, and test capabilities but no signature verification.
- [ ] Implement `GenericHmacProvider` using a configured signature header and shared secret reference; support verification and re-signing.
- [ ] Implement `StripeProvider` with verification, event type extraction, fixture naming, and fresh signature generation.
- [ ] Keep Stripe-specific behavior inside `StripeProvider`; no Stripe branches in generic capture, replay, or test generation services.
- [ ] Test each provider's capabilities, verification result, event extraction, fixture naming, and signing behavior.

## Task 5: Capture CLI

- [ ] Implement `php artisan preview:capture generic`.
- [ ] Implement `php artisan preview:capture hmac --signature-header=X-Signature`.
- [ ] Implement `php artisan preview:capture stripe`.
- [ ] Commands should open configured transport, print the public URL, and route inbound capture through provider-aware capture handling at `/__preview/capture/{provider}`.
- [ ] Tunnel capture should support open-print-close by default, `--wait` for an interactive session, and `--hold-seconds` for bounded smoke/demo windows.
- [ ] Add `--live` and reject live-mode capture unless config and CLI both explicitly allow it.
- [ ] Test command registration and command validation errors for unknown providers, missing signature header, and unsafe live mode.

## Task 5.5: HTTP Capture Endpoint

- [ ] Register a package route for `/__preview/capture/{provider}` when HTTP capture is enabled.
- [ ] Build `PreviewRequest` from Laravel's inbound request method, query, headers, and raw body.
- [ ] Store the request through `CaptureRepository` and `ProviderRegistry`.
- [ ] Respect `X-Preview-Original-Path` as the captured application path when present.
- [ ] Return safe JSON metadata: capture ID, provider, event type, verification status, and verification message.
- [ ] Return 404 JSON for unknown providers.
- [ ] Test generic capture, unknown provider, and sensitive header redaction.

## Task 6: Listing And Showing Captures

- [ ] Implement `php artisan preview:capture:list`.
- [ ] Implement `php artisan preview:capture:show {capture}`.
- [ ] List output should include capture ID, provider, event type when known, endpoint, verification status, and captured time.
- [ ] Show output should include redacted headers and metadata, never raw secrets.
- [ ] Test list/show output against stored captures.

## Task 7: Replay

- [ ] Implement `php artisan preview:capture:replay {capture} --exact`.
- [ ] Implement `php artisan preview:capture:replay {capture} --resign`.
- [ ] Implement optional `php artisan preview:capture:replay {capture} --exact --send-to=https://local-app.test`.
- [ ] Exact replay sends captured raw body and captured headers.
- [ ] Re-sign replay sends captured raw body and provider-fresh headers when `canSign()` is true.
- [ ] `--send-to` dispatches the replay payload over HTTP and reports status/success without mutating stored captures.
- [ ] If `--resign` is requested for a provider that cannot sign, fail with a clear message and suggest `--exact`.
- [ ] Test exact replay header/body fidelity, Stripe/HMAC re-signing, HTTP dispatch, and unsupported re-sign failure.

## Task 8: Fixture Generation

- [ ] Implement `php artisan preview:capture:fixture {capture}`.
- [ ] Write fixture metadata, redacted headers, raw payload reference, provider name, event type, endpoint, and signing mode.
- [ ] Generated committed fixture files must not contain authorization headers, cookies, or provider secrets.
- [ ] Generate raw payload files only when safe to commit or clearly mark them as local-only when not.
- [ ] Test fixture generation for generic, HMAC, and Stripe captures.

## Task 9: Pest Test Generation

- [ ] Implement `php artisan preview:capture:test {capture}`.
- [ ] Generate Pest tests that load the generated fixture through `PreviewFixture`.
- [ ] Generic test output uses captured headers.
- [ ] Stripe/HMAC signed test output uses fresh signed headers when available.
- [ ] Generated tests should run without manual edits when the target app has documented required state.
- [ ] If app state is not inferable, include an explicit precondition note in the generated test comments and keep the failing assertion meaningful.
- [ ] Test generated PHP syntax, expected output path, and provider-specific header behavior.

## Task 10: Acceptance Verification

- [x] Record current package acceptance evidence in `docs/preview/plans/v0.1-acceptance-evidence.md`.
- [ ] Create a fresh Laravel fixture app or Testbench scenario that installs the package as a dev dependency.
- [ ] Capture a generic inbound request and generate a fixture.
- [ ] Capture an HMAC-signed inbound request, verify it, replay it, and generate a Pest test.
- [ ] Capture a Stripe-style request, verify raw body fidelity, re-sign it, and generate Pest output.
- [ ] Verify `--exact` preserves captured headers/body.
- [ ] Verify `--resign` refreshes signatures where supported.
- [ ] Verify generated fixtures do not contain cookies, authorization headers, or provider secrets.
- [ ] Run the full package test suite and record the commands and output summary in the release notes or implementation PR.

## Done Criteria

v0.1 is complete only when:

- all v0.1 CLI commands exist
- generic, HMAC, and Stripe providers work through the same provider interface
- capture/replay/generate paths do not special-case Stripe outside its provider
- generated fixtures are reusable by generated Pest tests
- raw captures and secrets are local-first and protected by default
- acceptance verification passes in a supported Laravel app state
