# v0.3 Safe Route Preview Plan

## Target Command

```bash
php artisan preview:route {route} --ttl=2h --param=id=123 --session=currency=usd --readonly-db --guard=client
php artisan preview:route {route} --ttl=2h --param=id=123 --user-id=42 --user-model="App\Models\User" --guard=web
```

## Active v0.3 Slices

The active v0.3 slices move route preview from signed route-link metadata toward signed proxy execution, behavior-tested fakes, and explicit request context while keeping the safety boundary narrow and explicit.

### Slice A: Signed Proxy Execution

- named route lookup by `{route}`
- route method, URI, action, domain, and middleware summary
- route params from repeated `--param=key=value` flags
- session context from repeated `--session=key=value` flags, carried into the proxied preview request
- TTL signed access link generation
- guard context attached as request metadata from `--guard`
- default blocking for routes that do not allow `GET` or `HEAD`
- explicit opt-in before `POST`, `PUT`, `PATCH`, or `DELETE` routes can be exposed
- signed preview links proxy and execute the named route
- `--readonly-db` transaction wrapper for covered database writes inside the wrapped request
- behavior-tested side-effect fake flags for supported queue, mail, HTTP, and event facades

### Slice B: Hardening And Audit Output

- fail clearly when required route parameters are missing
- handle optional route parameters without forcing placeholder values
- support domain parameters in link generation and execution
- test signed-link expiry and signature rejection paths
- print clearer warnings for non-read methods, uncovered side effects, and partial readonly coverage
- emit audit output with route name, method, URI/domain, params, session context keys, guard metadata, TTL, safety flags, and execution mode

### Slice C: Behavior-Tested Mail Fake Coverage

- verify `--fake-mail` applies Laravel's mail fake during proxied route execution
- cover a route that sends mail so the test proves behavior, not only audit text
- keep the warning boundary explicit: mail faking does not isolate queue, event, cache, filesystem, external HTTP, or database writes

### Slice D: App-Specific Auth Context

- accept `--user-id={id}` for routes that need an authenticated application user
- accept optional `--user-model=App\Models\User` when the app cannot rely on the default authenticatable model
- keep `--guard` as guard selection and request/audit metadata
- authenticate through the host Laravel app's normal guard and authenticatable model behavior
- fail clearly when the user is missing, the model class is not authenticatable, or the selected guard cannot authenticate the resolved user
- do not claim generic authorization bypass, policy bypass, middleware bypass, or full scenario isolation

## Safety Contract

`--readonly-db` is the flag name. Do not introduce `--readonly`.

`--readonly-db` is not a complete read-only guarantee. It only wraps the covered preview request in a database transaction so database writes inside that request can be rolled back. It does not prove that route execution cannot mutate application state outside the database transaction.

Database rollback will not protect:

- queued jobs
- sent mail
- cache writes
- filesystem writes
- external HTTP calls
- emitted events with side effects
- code that uses separate database connections outside the wrapper

Route preview does not provide cache isolation or filesystem isolation.

`--guard` alone does not impersonate users or authenticate the request as a specific user. Authenticated preview must be explicit through the app-specific `--user-id` slice, with optional `--user-model` when required. `--guard` remains guard selection and request/audit metadata; it is not a generic authorization bypass.

Use explicit fake flags for supported side effects when the preview must suppress more than database writes:

```bash
--fake-queue
--fake-mail
--fake-http
--fake-events
```

## Implementation Notes

1. Resolve `{route}` through Laravel's named route collection.
2. Fail clearly when the named route does not exist.
3. Render a middleware summary before printing any access link.
4. Require all mandatory route parameters to be provided through `--param`.
5. Sign generated access links with the requested TTL.
6. Allow optional route parameters to be omitted when Laravel can generate the route without them.
7. Parse repeated `--session=key=value` flags and carry that context into the proxied preview request session.
8. Include domain parameters in validation, signing, audit output, and execution.
9. Block routes that do not advertise `GET` or `HEAD` unless the explicit non-read method opt-in is present.
10. Execute the target route through the signed preview proxy.
11. Treat `--guard` as preview request metadata unless the app-specific auth context slice exists and is tested.
12. Wrap covered route execution in a database transaction when `--readonly-db` is present.
13. Apply only the explicit side-effect fakes requested by flag.
14. Behavior-test supported fake flags against the relevant Laravel facades, not only command/audit output.
15. Add a mail-sending route test that proves `--fake-mail` suppresses mail during proxied execution.
16. Test expiry, invalid signatures, missing required params, optional params, domain params, session context, unsafe-method warnings, and audit output.
17. Add app-specific auth context using `--user-id`, optional `--user-model`, and selected `--guard` without claiming policy bypass or full scenario isolation.

## Follow-Up Work

Later v0.3 work can add cache/filesystem protections if they become practical and broader fake coverage. Those follow-ups should remain separate from the signed proxy execution and app-specific auth context work so the current safety boundary stays truthful.
