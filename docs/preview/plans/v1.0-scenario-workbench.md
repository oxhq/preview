# v1.0 Scenario Workbench Plan

## Product Frame

Scenario Workbench keeps Laravel Preview focused on local-first Laravel flow reproduction. A scenario should describe a reusable local application flow by composing existing captures, named routes, fakes, optional seeded state, and notes. It should not turn the package into a hosted workflow runner or generic tunnel manager.

## Foundation Slice

The first slice is intentionally read-only:

- local PHP scenario files under `preview/scenarios` by default
- configurable path through `preview.scenario_path` / `PREVIEW_SCENARIO_PATH`
- each file returns `Oxhq\Preview\Scenario\Scenario`
- scenario repository loads and validates local files
- `preview:scenario:list` lists available local scenarios
- `preview:scenario:show {scenario}` prints scenario details

Scenario fields:

- `name`: stable local name used by commands
- `captures`: local capture IDs
- `routes`: Laravel route names
- `routeParameters`: optional route parameters keyed by route name
- `routeContext`: optional per-route execution context keyed by route name; supports `session`, `guard`, `user_id`, `user_model`, `readonly_db`, and route-specific `fakes`
- `fakes`: supported fake boundaries such as queue, mail, HTTP, and events
- `seed`: optional Laravel seeder class
- `notes`: optional operator context

Example:

```php
use App\Database\Seeders\DemoSubscriptionSeeder;
use Oxhq\Preview\Scenario\Scenario;

return new Scenario(
    name: 'subscription-renewal',
    seed: DemoSubscriptionSeeder::class,
    routes: ['billing.portal'],
    routeParameters: [
        'billing.portal' => ['id' => '123'],
    ],
    routeContext: [
        'billing.portal' => [
            'session' => ['tenant' => 'acme'],
            'guard' => 'web',
            'readonly_db' => true,
            'fakes' => ['events'],
        ],
    ],
    captures: ['20260505011852323-sugxujb2'],
    fakes: ['queue', 'mail'],
    notes: 'Exercises the local renewal review flow after a captured provider callback.',
);
```

## Explicit Non-Goals For Foundation

Do not claim these are complete in the foundation slice:

- scenario replay
- seeder execution or seeded-state composition
- route-preview execution composition
- capture replay orchestration
- scenario test generation
- CI replay
- hosted scenario sharing

## Seed And Capture Replay Slices

These slices add the first executable scenario behavior after the read-only foundation:

- `preview:scenario:replay {scenario}`
- `--exact` for stored raw body plus captured headers
- `--resign` for stored raw body plus fresh provider-valid signature headers where supported
- optional seed execution through Laravel's normal seeder path before replay
- optional `--send-to={url}` dispatch through the shared replay dispatcher
- clear failure output for missing scenarios, missing captures, unsupported resign, and failing seeders

## Acceptance Evidence

The foundation slice is done only when the repo proves:

- scenario files can be loaded from the configured local path
- invalid files fail clearly when they do not return `Scenario`
- empty names fail clearly
- list output is sorted and includes scenario names
- show output includes captures, route names, fakes, seed class, and notes
- docs do not claim replay, seeded composition, or generated tests beyond implemented behavior

## Later Slices

1. Route composition: execute named route previews from `preview:scenario:replay` with the same safety flags and boundaries documented for route preview.
2. Fake propagation: pass scenario `fakes` into route-preview execution and prove the supported queue, mail, HTTP, and event fakes apply only within those documented Laravel fake boundaries.
3. Test generation: generate Pest-compatible scenario tests after seed, capture replay, route composition, and fake propagation have package tests.

Each later slice needs its own package tests and, before public claims, a fresh Laravel consumer smoke.

## Replay Contract

When a scenario declares `seed`, replay runs that seeder through Laravel's normal seeder path before captures execute. A missing, invalid, or failing seeder stops the scenario with clear output; it is not treated as optional metadata once replay is requested.

The command shape is:

```bash
php artisan preview:scenario:replay subscription-renewal --exact
php artisan preview:scenario:replay subscription-renewal --resign
```

Replay semantics:

- `--exact` replays every listed capture with its stored raw body and captured headers.
- `--resign` replays every listed capture with its stored raw body and fresh provider-valid signature headers when the provider supports signing.
- unsupported re-signing, missing captures, and provider lookup failures fail clearly instead of silently skipping or falling back.
- seed execution runs before capture replay when a seed is configured.

Route composition is package-local proof once named route previews execute through scenario replay under the route-preview safety flags and documented side-effect boundaries. Scenario route-context composition is package-local proof once package tests show per-route session, guard, user, readonly database, and fake context reaching route-preview execution without changing route module APIs. Scenario fake propagation is package-local proof once package tests show scenario `fakes` reaching the route-preview request and applying the supported Laravel fakes during route execution. Scenario test generation is package-local proof once Pest-compatible scenario test files are generated from real scenario files, linted with the local PHP binary, and exercised through a package-local Pest/Laravel command-chain harness. That still is not consumer proof.

## Scenario Test Generation Contract

Target output should be Pest-compatible and local-first:

- load a real scenario file by name
- run the configured seeder before replay
- replay listed captures in the requested mode
- execute composed route previews only when that slice is already proven
- preserve explicit precondition comments for app state, auth context, route parameters, and fake boundaries

Generated scenario tests should fail clearly when the target Laravel app lacks the required route, model, database state, provider secret, or fake support. Do not generate tests that hide missing app state behind broad mocks.

## Evidence Boundaries

Package tests may prove local package behavior for scenario replay, route composition, fake propagation, and generated scenario test output. They do not prove:

- fresh consumer app compatibility unless a clean Laravel app installs the package and runs the same workflow
- hosted CI unless a real hosted check passes on the target branch or PR
- Packagist installation unless the package is installed from Packagist
- live tunnel startup unless real cloudflared/ngrok binaries are exercised
- SaaS replay, persistent URLs, managed relay, team sharing, or audit logs

## Remaining SaaS And CI Boundaries

Keep SaaS and CI replay out of v1.0 until there is direct demand and proof:

- CI replay requires generated scenario assets that run predictably in hosted checks.
- Cloud replay requires hosted request execution through a real URL.
- Team sharing requires access control, storage ownership, audit logs, and revocation.
- Persistent URLs require expiry, abuse controls, and secret handling.
- Managed relay requires operational monitoring and failure reporting.
